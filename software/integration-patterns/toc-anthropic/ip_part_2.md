# Integration Patterns Complete Study Guide - Part 2: Message Construction & Routing

## 3.0 Message Construction

### 3.1 Message Types
#### 3.1.1 Semantic Message Types
- 3.1.1.1 Command message — instruction to do something — imperative — CreateOrder / SendEmail
  - 3.1.1.1.1 One receiver — point-to-point semantics — only one system executes the command
  - 3.1.1.1.2 Request/reply pair — CommandMessage → ReplyMessage — synchronous feel over async
- 3.1.1.2 Event message — notification that something happened — past tense — OrderPlaced
  - 3.1.1.2.1 Pub/sub semantics — many listeners — producer unaware of consumers — decoupled
  - 3.1.1.2.2 Thin event — only ID + timestamp — consumers fetch details — avoids data duplication
  - 3.1.1.2.3 Fat event — full state included — consumers self-contained — no extra fetch needed
- 3.1.1.3 Document message — transfer a data structure — no instruction — just information
  - 3.1.1.3.1 Report / sync — pass document between systems — content drives processing

### 3.2 Message Structure
#### 3.2.1 Envelope Pattern
- 3.2.1.1 Header + body — metadata in header — business payload in body — separation of concerns
  - 3.2.1.1.1 Headers — message-id / correlation-id / timestamp / content-type / reply-to
  - 3.2.1.1.2 Routing decisions based on headers — body opaque to infrastructure — clean layering
- 3.2.1.2 Correlation identifier — unique ID linking request to its reply — async traceability
  - 3.2.1.2.1 UUID generated by requester — echoed in reply — match on return — essential pattern
  - 3.2.1.2.2 Distributed tracing — trace-id propagated — spans correlated — OpenTelemetry W3C
- 3.2.1.3 Return address — reply-to header — tells receiver where to send response — decoupled
  - 3.2.1.3.1 Dynamic reply queues — temporary queue per request — cleaned up after reply
- 3.2.1.4 Message sequence — sequence number + size — large message split across parts — ordered
  - 3.2.1.4.1 Sequence header — sequence-number / sequence-size — aggregator reassembles

### 3.3 Message Metadata Patterns
#### 3.3.1 Identity & Expiry
- 3.3.1.1 Message ID — globally unique — UUID v4 or v7 — deduplication key — idempotency
  - 3.3.1.1.1 Idempotent consumer stores processed IDs — duplicate message = skip processing
- 3.3.1.2 Expiration / TTL — message invalid after timestamp — consumer checks before processing
  - 3.3.1.2.1 Stale message detection — event-time vs processing-time — discard outdated commands
- 3.3.1.3 Format indicator — content-type header — application/json / avro / protobuf / xml
  - 3.3.1.3.1 Schema version — X-Schema-Version header or Avro schema ID — consumer knows how to parse

## 4.0 Message Routing

### 4.1 Basic Routers
#### 4.1.1 Content-Based Router
- 4.1.1.1 Examine message content — route to appropriate channel — single correct destination
  - 4.1.1.1.1 Switch on message type — command.type == "OrderPlaced" → orders channel
  - 4.1.1.1.2 Switch on content field — order.region == "EU" → EU processing queue
  - 4.1.1.1.3 Avoid business logic in router — keep routing rules simple — maintainability

#### 4.1.2 Message Filter
- 4.1.2.1 Discard messages not meeting criteria — downstream only sees relevant messages
  - 4.1.2.1.1 Declarative filter rules — SQL-like predicates — AWS SNS filter policies — efficient
  - 4.1.2.1.2 Consumer-side filter — receive all — ignore irrelevant — wasteful but simpler

### 4.2 Advanced Routers
#### 4.2.1 Splitter
- 4.2.1.1 Split composite message into individual messages — process each independently
  - 4.2.1.1.1 Order with line items — split to one message per item — parallel processing
  - 4.2.1.1.2 Maintain sequence info — each split message carries sequence-number + size — for later aggregation
- 4.2.1.2 Recipient list — route message to dynamically computed set of recipients
  - 4.2.1.2.1 Dynamic list — compute recipients from message content or lookup — flexible fanout
  - 4.2.1.2.2 Each-recipient-once — deduplication — avoid double-processing — idempotent

#### 4.2.2 Aggregator
- 4.2.2.1 Collect related messages — combine into single message — inverse of splitter
  - 4.2.2.1.1 Correlation key — group messages by shared ID — collect all splits — match group
  - 4.2.2.1.2 Completion condition — all N parts received / timeout / first response — trigger output
  - 4.2.2.1.3 Stateful — aggregator must store partial state — persistent store — failure recovery

#### 4.2.3 Scatter-Gather
- 4.2.3.1 Broadcast request to multiple recipients — collect all replies — aggregate result
  - 4.2.3.1.1 Parallel fan-out — all requests sent simultaneously — reduce total latency
  - 4.2.3.1.2 Best-of-N — return first/best reply — auction pattern — price comparison
  - 4.2.3.1.3 Timeout policy — wait N ms for replies — proceed with what arrived — partial results

#### 4.2.4 Resequencer
- 4.2.4.1 Out-of-order messages — buffer — reorder by sequence number — pass in order
  - 4.2.4.1.1 Sequence gap detection — hold messages until gap filled — or timeout + emit gap
  - 4.2.4.1.2 Kafka ordering guarantee — within partition — partition key ensures order per entity

#### 4.2.5 Process Manager
- 4.2.5.1 Stateful orchestrator — routes messages through multi-step process — tracks state
  - 4.2.5.1.1 Replaces chains of routers — single responsible coordinator — clearer flow
  - 4.2.5.1.2 State persistence — survives restarts — durable — DB-backed or event-sourced
  - 4.2.5.1.3 Relates to Saga pattern — process manager = orchestrated saga — central control
