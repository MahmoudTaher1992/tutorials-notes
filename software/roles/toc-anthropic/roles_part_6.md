# Software Company Roles Study Guide - Part 6: Senior Leadership, Cross-functional & Hiring

## 9.2 Staff / Principal / Distinguished Engineer
### 9.2.1 The Senior IC Track
#### 9.2.1.1 Why Senior IC Track Exists
- 9.2.1.1.1 Not all great engineers want to manage — IC track allows continued growth
  - 9.2.1.1.1.1 IC track equivalent in authority — Staff = EM / Principal = Director
  - 9.2.1.1.1.2 Different leverage — ICs multiply through code + design + standards
- 9.2.1.1.2 Glue work vs. technical work — ICs do both — not just code
  - 9.2.1.1.2.1 Writing — RFCs / design docs / ADRs — durable institutional knowledge
  - 9.2.1.1.2.2 Meetings — make technical decisions happen — cross-team alignment

#### 9.2.1.2 Staff Engineer Archetypes (Will Larson)
- 9.2.1.2.1 Tech Lead — lead team-level technical work — most common archetype
- 9.2.1.2.2 Architect — cross-cutting technical direction — system design ownership
- 9.2.1.2.3 Solver — fix hard ambiguous problems — roving assignment — firefighter
- 9.2.1.2.4 Right Hand — extend EM/Director reach — execution of strategic initiatives

## 9.3 VP of Engineering
### 9.3.1 Core Responsibilities
- 9.3.1.1 Lead engineering organization — multiple teams — org design + culture + delivery
  - 9.3.1.1.1 Org design — team structure + reporting lines + headcount planning
  - 9.3.1.1.2 Engineering velocity — DORA metrics — track + improve delivery performance
- 9.3.1.2 Strategy execution — translate company strategy into engineering roadmap
  - 9.3.1.2.1 Resource allocation — headcount + budget — prioritize by business impact
  - 9.3.1.2.2 Build vs. buy decisions — framework — cost + time + strategic value
- 9.3.1.3 Stakeholder management — CTO + CPO + CEO + Board — represent engineering
  - 9.3.1.3.1 Executive communication — status + risks + asks — concise — decision-ready
- 9.3.1.4 VPE vs. CTO distinction
  - 9.3.1.4.1 VPE — runs the organization — people + delivery + ops — internal focus
  - 9.3.1.4.2 CTO — external + technical vision — customer + product + strategy focus

## 9.4 Chief Technology Officer (CTO)
### 9.4.1 CTO Archetypes
- 9.4.1.1 Startup CTO — hands-on — writes code — first engineer — sets foundations
  - 9.4.1.1.1 Technical co-founder — builds MVP — hires first team — defines culture
  - 9.4.1.1.2 Transitions — as company grows — move from IC to leader — hard shift
- 9.4.1.2 Scaleup CTO — technology evangelist — external voice — product visionary
  - 9.4.1.2.1 External representation — conferences + press + customers — trust builder
  - 9.4.1.2.2 Partnership + vendor strategy — technology ecosystem — build vs. integrate
- 9.4.1.3 Enterprise CTO — governance + compliance + architecture standards — large org

### 9.4.2 CTO Core Responsibilities
- 9.4.2.1 Technology vision — 3–5 year horizon — where technology is going
  - 9.4.2.1.1 Horizon scanning — emerging tech — build conviction — experiment early
  - 9.4.2.1.2 Technology radar — track + categorize — adopt/trial/assess/hold
- 9.4.2.2 Board + investor communication — technical due diligence — architecture review
- 9.4.2.3 Engineering culture — values + principles — attract + retain top talent
  - 9.4.2.3.1 Employer brand — engineering blog + OSS + talks — reputation as magnet

## 10.0 Cross-functional Roles

### 10.1 Technical Writer
#### 10.1.1 Core Responsibilities
- 10.1.1.1 Create + maintain technical documentation — API docs + guides + references
  - 10.1.1.1.1 Docs as code — Markdown + Git — reviewed like code — versioned
  - 10.1.1.1.2 API documentation — OpenAPI / AsyncAPI — auto-generated + hand-written
- 10.1.1.2 Developer experience — clear onboarding — reduce time to first success
  - 10.1.1.2.1 Tutorial — task-oriented — step by step — beginner focus
  - 10.1.1.2.2 How-to guide — goal-oriented — assumes knowledge — intermediate
  - 10.1.1.2.3 Reference — information-oriented — complete — advanced
  - 10.1.1.2.4 Explanation — understanding-oriented — concepts — background
- 10.1.1.3 Collaborates with engineers — review for accuracy — engineers own content

---

### 10.2 Developer Advocate / Developer Relations (DevRel)
#### 10.2.1 Core Responsibilities
- 10.2.1.1 Represent developers inside company — voice of the external developer
  - 10.2.1.1.1 Feedback loop — bring developer pain points to product team
  - 10.2.1.1.2 Product feedback — which APIs are confusing — what blocks adoption
- 10.2.1.2 Represent company to developers — content + talks + demos + community
  - 10.2.1.2.1 Conference talks — technical depth — credibility — not sales pitch
  - 10.2.1.2.2 Sample code + tutorials — show best practices — reduce time to value
- 10.2.1.3 Community building — Discord / Slack / forum — active + welcoming
  - 10.2.1.3.1 Developer champions program — top community members — amplify reach

---

### 10.3 Solutions Architect
#### 10.3.1 Core Responsibilities
- 10.3.1.1 Bridge between product capability and customer technical needs
  - 10.3.1.1.1 Pre-sales — discover customer requirements — design solution — demo
  - 10.3.1.1.2 Technical validation — prove solution works — POC with customer
- 10.3.1.2 Post-sales — implementation guidance — best practices — success path
  - 10.3.1.2.1 Architecture review — validate customer design — prevent anti-patterns
  - 10.3.1.2.2 Escalation path — customer technical issues → engineering team
- 10.3.1.3 Feedback to product — common customer problems — feature gaps — input to roadmap

---

### 10.4 Scrum Master / Agile Coach
#### 10.4.1 Scrum Master Responsibilities
- 10.4.1.1 Serve the team — remove impediments — protect from outside interruption
  - 10.4.1.1.1 Impediment tracking — blocker log — escalate if not resolved in 24h
  - 10.4.1.1.2 Meeting facilitation — daily standup + planning + review + retro
- 10.4.1.2 Coach on Agile practices — guide not enforce — build team self-sufficiency
  - 10.4.1.2.1 Retrospective formats — Start/Stop/Continue — 4Ls — Mad/Sad/Glad
  - 10.4.1.2.2 Velocity tracking — not target — diagnostic tool — predictability
- 10.4.1.3 Scrum Master ≠ Project Manager — no authority over team — servant leader

## 11.0 Hiring & Career Ladders

### 11.1 Leveling Frameworks
#### 11.1.1 IC vs. Management Track
- 11.1.1.1 IC track — Individual Contributor — L1 through Distinguished
  - 11.1.1.1.1 Promotion criteria — scope + impact + behaviors — not tenure
  - 11.1.1.1.2 Promo packet — evidence of impact — written by IC + manager
- 11.1.1.2 Management track — EM → Senior EM → Director → VP → CTO
  - 11.1.1.2.1 Track switch — IC → management — reversible — both respected
- 11.1.1.3 Calibration sessions — cross-team leveling — prevent local grade inflation
  - 11.1.1.3.1 Calibration committee — peers + skip-levels — consistent bar across org

### 11.2 Interview & Hiring Process
#### 11.2.1 Structured Hiring
- 11.2.1.1 Job description — clear scope + level + impact — attract right candidates
- 11.2.1.2 Interview loop — defined rounds — each assesses different dimension
  - 11.2.1.2.1 Technical screen — coding or system design — 45–60 min
  - 11.2.1.2.2 Behavioral — STAR method — past behavior predicts future — 45 min
  - 11.2.1.2.3 System design — architecture — scale — ambiguous problem — 60 min
  - 11.2.1.2.4 Culture add — values alignment — not culture fit — diversity lens
- 11.2.1.3 Debrief — structured — each interviewer submits written feedback first
  - 11.2.1.3.1 Hire / No-hire / Strong hire — no maybes in final decision
  - 11.2.1.3.2 Bar raiser — independent interviewer — ensures quality bar maintained
