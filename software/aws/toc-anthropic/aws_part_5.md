# AWS Complete Study Guide - Part 5: Security & Identity (Phase 1 — Ideal)

## 5.0 Security & Identity

### 5.1 Identity & Access Management
#### 5.1.1 IAM Principal Types
- 5.1.1.1 IAM Users — long-term credentials — access key + secret — avoid for apps
  - 5.1.1.1.1 Access key rotation — 90-day policy — IAM Credential Report
  - 5.1.1.1.2 Console password policy — min length, complexity, reuse prevention
- 5.1.1.2 IAM Roles — temporary credentials via STS — preferred for applications
  - 5.1.1.2.1 EC2 Instance Profile — attach role to EC2 — IMDS credential delivery
  - 5.1.1.2.2 Service-linked roles — pre-defined permissions for AWS services
- 5.1.1.3 IAM Groups — collection of users — attach policies to group not user
- 5.1.1.4 Federated Identities — SAML 2.0 / OIDC — external IdP (Okta, Azure AD)

#### 5.1.2 Policy Types & Evaluation
- 5.1.2.1 Identity-based policies — managed (AWS/customer) vs. inline
  - 5.1.2.1.1 AWS Managed policies — broadly applicable — updated by AWS
  - 5.1.2.1.2 Customer Managed — version-controlled — up to 5 versions retained
- 5.1.2.2 Resource-based policies — inline on resource (S3 bucket policy, SQS)
  - 5.1.2.2.1 Cross-account without role assumption — resource policy + STS
- 5.1.2.3 Permission boundaries — max permissions ceiling — delegate admin safely
  - 5.1.2.3.1 Effective permissions = identity policy ∩ permission boundary
- 5.1.2.4 SCPs (Service Control Policies) — AWS Organizations — deny hierarchy
  - 5.1.2.4.1 SCP doesn't grant — only restricts — must also have identity policy
- 5.1.2.5 Session policies — inline during AssumeRole — further restrict session

#### 5.1.3 Policy Evaluation Logic
- 5.1.3.1 Evaluation order — explicit Deny → SCP → resource policy → identity policy
  - 5.1.3.1.1 Explicit Deny always wins — overrides any Allow
  - 5.1.3.1.2 Implicit Deny by default — no matching Allow = deny
- 5.1.3.2 Cross-account evaluation — both sides must allow
  - 5.1.3.2.1 Resource policy Allow + caller's identity policy Allow = access

### 5.2 Authentication Mechanisms
#### 5.2.1 AWS STS (Security Token Service)
- 5.2.1.1 AssumeRole — returns AccessKeyId, SecretAccessKey, SessionToken
  - 5.2.1.1.1 Session duration — 15min to 12hrs — default 1hr — role max set
  - 5.2.1.1.2 External ID — prevent confused deputy — third-party cross-account
- 5.2.1.2 AssumeRoleWithWebIdentity — OIDC token exchange — IRSA for EKS
  - 5.2.1.2.1 IRSA — IAM Roles for Service Accounts — per-pod AWS credentials
- 5.2.1.3 AssumeRoleWithSAML — enterprise SSO — SAML assertion exchange
- 5.2.1.4 GetFederationToken — long-term cred holder creates federated session

#### 5.2.2 Amazon Cognito
- 5.2.2.1 User Pools — OIDC IdP — JWT tokens (ID/Access/Refresh)
  - 5.2.2.1.1 Hosted UI — pre-built OAuth2 login page — custom domain support
  - 5.2.2.1.2 User migration Lambda trigger — import on first sign-in
  - 5.2.2.1.3 Adaptive authentication — risk-based MFA — ML-detected anomalies
- 5.2.2.2 Identity Pools — exchange any token for temporary AWS credentials
  - 5.2.2.2.1 Authenticated vs. unauthenticated roles — guest access pattern
  - 5.2.2.2.2 Enhanced flow — automatic role selection based on token claims

### 5.3 Key Management (KMS)
#### 5.3.1 KMS Key Types
- 5.3.1.1 AWS Owned Keys — free — AWS manages entirely — no visibility
- 5.3.1.2 AWS Managed Keys (aws/service) — free — per-service — CloudTrail visible
- 5.3.1.3 Customer Managed Keys (CMK) — $1/month — full control — rotation policy
  - 5.3.1.3.1 Symmetric CMK (AES-256-GCM) — single key for encrypt/decrypt
  - 5.3.1.3.2 Asymmetric CMK — RSA/ECC — encrypt or sign — public key exportable
  - 5.3.1.3.3 HMAC CMK — generate/verify MACs — token validation

#### 5.3.2 Key Policies & Grants
- 5.3.2.1 Key policy — resource-based — controls who can use and administer key
  - 5.3.2.1.1 Key policy must explicitly allow account root — no implicit access
  - 5.3.2.1.2 Principal: "*" with condition — cross-service delegation
- 5.3.2.2 Grants — temporary delegated access — programmatic — per-operation
  - 5.3.2.2.1 EncryptionContext — additional AAD — must match on decrypt
  - 5.3.2.2.2 Retiring principal — auto-retire grants — Lambda temp access

#### 5.3.3 Envelope Encryption
- 5.3.3.1 Data Encryption Key (DEK) — generated by KMS — encrypts actual data
  - 5.3.3.1.1 GenerateDataKey — plaintext DEK + ciphertext DEK returned
  - 5.3.3.1.2 Store ciphertext DEK with data — discard plaintext after encrypt
- 5.3.3.2 Decrypt flow — KMS.Decrypt(ciphertext DEK) → plaintext DEK → decrypt data
- 5.3.3.3 Key rotation — annual auto-rotation — backing key rotated — old data readable

#### 5.3.4 CloudHSM
- 5.3.4.1 Dedicated hardware HSM — FIPS 140-2 Level 3 — single-tenant
  - 5.3.4.1.1 Cluster HA — minimum 2 HSMs in different AZs
  - 5.3.4.1.2 PKCS#11 / JCE / CNG APIs — bring existing HSM applications

### 5.4 Secrets Management
#### 5.4.1 AWS Secrets Manager
- 5.4.1.1 Automatic rotation — Lambda-backed — DB credentials rotated without downtime
  - 5.4.1.1.1 Rotation window — rotation frequency 1 day to 365 days
  - 5.4.1.1.2 Multi-user rotation strategy — create new → test → delete old
- 5.4.1.2 Cross-account access — resource policy on secret — STS cross-account
- 5.4.1.3 VPC endpoint — access secrets without internet — PrivateLink

#### 5.4.2 AWS Systems Manager Parameter Store
- 5.4.2.1 Standard parameters — free — 10KB — no rotation
- 5.4.2.2 Advanced parameters — $0.05/month — 8KB — TTL expiry policies
  - 5.4.2.2.1 SecureString — KMS-encrypted — use CMK or aws/ssm
  - 5.4.2.2.2 Hierarchies — /prod/db/password — IAM path-based restriction

### 5.5 Network Security
